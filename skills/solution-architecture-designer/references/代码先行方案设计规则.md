# 代码先行方案设计规则

## 1. 定位

软件方案设计是编码前的轻量实现路线决策，不是详细设计。它必须基于用户指定本地代码目录的代码分析结果，说明如何用最小改动承接需求。

权威详细设计应在代码实现并测试通过后，由后续详细设计阶段基于真实代码回填。

## 2. 输入约束

- 必须有用户指定的本地代码目录，或已有 `state/structured/codebase-analysis.json`。
- 方案设计不直接扫描代码，不直接替代 `codebase-analysis-reporter`。
- 代码事实只来自 `codebase-analysis.json`、代码分析报告或用户明确提供的代码依据。
- 缺少代码事实时，记录阻塞项，不生成正式方案。

## 3. 复用优先

推荐方案必须优先考虑：

- 复用现有模块。
- 复用现有接口。
- 复用现有数据结构。
- 复用现有配置/标定机制。
- 复用现有测试资产。

如果不能复用，必须说明现有代码为什么无法承接，并给出代码依据或待确认项。

## 4. 最小改动

方案应说明：

- 哪些代码不改。
- 哪些模块小改。
- 哪些能力新增。
- 哪些旧接口保持兼容。
- 哪些风险通过测试验证。

禁止先提出全新架构，再倒推代码如何适配。

## 5. 代码依据

以下内容必须带代码依据：

- 现有能力复用。
- 影响模块。
- 接口与数据影响。
- 推荐方案中关于现有代码的确定判断。

没有代码依据的内容只能写入推断项或待确认项。

## 6. 不做详细设计

方案设计不得展开为函数、类、字段、文件级详细设计。除非这些对象已经存在于代码分析事实中，并且只是说明“现有对象会被复用或受影响”。

禁止出现：

- 新增某个具体类作为方案核心。
- 新增某个具体函数签名。
- 设计具体字段结构。
- 规定具体文件内部实现。

这些内容应留到代码实现或编码后详细设计阶段。

## 7. 验证策略

每个主要风险都应有验证方式，可选：

- 评审
- 静态分析
- 单元测试
- 集成测试
- 台架测试
- HIL测试
- 实车测试
- 待确认
