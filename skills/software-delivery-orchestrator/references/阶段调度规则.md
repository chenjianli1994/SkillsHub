# 阶段调度规则

## 基本原则

- 总工作流只做编排，不替代阶段 skill。
- 阶段 skill 不存在时，记录阻塞项，不伪造产物。
- 用户可见产物写入 `outputs/`。
- 内部 JSON、状态文件和画像写入 `state/`。

## 缺失 skill 处理

| 缺失项 | 处理方式 |
| --- | --- |
| `solution-architecture-designer` | 记录“方案设计 skill 尚未建立”，不生成方案文档 |
| `detailed-design-writer` | 记录“详细设计 skill 尚未建立”，不生成详细设计 |
| `implementation-task-planner` | 记录“任务计划 skill 尚未建立”，不生成任务计划 |
| `software-quality-gate` | 使用已有质量检查 skill 可覆盖的部分，并记录缺口 |

## 需求拆解阶段调度

需求拆解由 `requirement-decomposition-planner` 承担，采用 ASPICE SWE.1 风格的软件需求分析方式。

- 输入：优先读取 `outputs/02_需求规格书.md` 和 `state/structured/requirement-specification.json`；也可接受客户需求、系统需求或用户补充的上游需求材料。
- 输出：`outputs/03_需求拆解清单.md` + `state/structured/requirement-breakdown.json`。
- 核心视图：软件需求清单（`SWR-*`）、CM 符合性矩阵、覆盖性统计、一致性/不完整性问题、设计输入集。
- 缺少上游需求材料时：记录阻塞项，不伪造软件需求。
- 本阶段不替代开发任务计划，不输出前端/后端/数据库等开发任务拆分。

## 代码分析阶段调度

现有代码分析由 `codebase-analysis-reporter` 承担，是代码驱动模式（`design-from-code` / `strategy-from-code` / `codebase-only`）的事实来源起点。

- 入口 `codebase-only`：只做代码分析，产出 `04_现有代码分析报告.md` + `codebase-analysis.json`。
- 代码驱动链路：先 `codebase-analysis-reporter`，再 `solution-architecture-designer` / `detailed-design-writer` / `strategy-document-writer`；下游读 `state/structured/codebase-analysis.json`，不直接读代码。
- 代码生成闭环：`implementation-executor` 测试通过后，`codebase-analysis-reporter` 以「测试通过的代码 + implementation-record.json」为事实来源，供下游回填权威版详设/策略。
- 三态强制：分析结论必须区分确定事实（给代码依据）、合理推断（标记推断）、待确认项，不得把推断写成事实。

## 输出边界

`outputs/` 只能放用户需要的产出物。以下内部产物不得写入 `outputs/`：

- 工作流状态 JSON
- 结构化阶段 JSON
- 参考文档画像 JSON
- `state/` 下的内部画像 Markdown
- 内部调度记录

如果用户明确要求查看参考文档画像，应生成面向评审的独立 Markdown 交付物，不要把 `state/` 内部文件直接复制到 `outputs/`。

## 代码实现阶段调度

代码实现由 `implementation-executor` 承担，位于任务计划（阶段8）之后、代码驱动回填文档之前。

- 入口 `implement-from-plan`：有 `08_开发任务计划.md` 或 `06_软件详细设计说明书.md` 即可执行；两者都缺时降级基于 `02_需求规格书.md`，并记录「依据不足」阻塞项。
- 入口 `full-flow-with-code`：全流程做到代码——需求分析 → …… → 任务计划 → 代码实现 → 代码驱动回填详设/策略 → 质量门禁。
- 代码写入代码仓库/工作区，**不写入 `outputs/`**；实现记录写入 `state/structured/implementation-record.json`；执行报告写入 `outputs/09_开发执行报告.md`。
- 代码实现完成后，只有实现记录中 `可进入代码驱动出文档 = true`（全部任务测试通过、无阻塞）时，才触发 `codebase-analysis-reporter` → `detailed-design-writer` / `strategy-document-writer`，以真实代码反出权威版详设/策略。
- 详设/策略文档的权威版本以「编码后测试通过」的这一版为准，编码前的前向设计只作为实现输入。

`implementation-executor` 依赖的上游 skill（`implementation-task-planner`、`detailed-design-writer`、`solution-architecture-designer`）尚未建立时，按上方「缺失 skill 处理」记录阻塞，可基于已有产物降级执行，不伪造前置设计。
