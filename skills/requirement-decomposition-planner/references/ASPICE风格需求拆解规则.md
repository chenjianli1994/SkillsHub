# ASPICE 风格需求拆解规则

## 1. 定位

需求拆解阶段参考 ASPICE SWE.1 的软件需求分析思想：从上游客户/系统/规格书需求形成软件需求基线，确保软件需求清晰、一致、可验证、可追踪，并能作为软件方案设计、详细设计和验证策划的输入。

本阶段不是开发任务计划，不输出前端、后端、数据库、接口开发等任务分工。

## 2. 上游需求基线

- 保留上游已有编号，如 `SYS-*`、`FR-*`、`NFR-*`、`DR-*`、`PR-*`。
- 无编号但可作为需求来源的条目，生成 `UR-*` 代理编号。
- 每条上游需求应记录类型、描述、来源、成熟度和备注。
- 上游需求不清晰时，不要擅自补全为确定需求，写入一致性问题或待确认项。

## 3. 软件需求拆解

软件需求使用 `SWR-*` 编号。每条软件需求必须满足：

- 能追溯到至少一个上游需求。
- 描述软件应当具备的能力、行为、约束或可验证属性。
- 不固化具体代码实现，除非上游明确要求。
- 能给出验证方式。
- 能进入后续软件方案设计和详细设计。

允许的需求类型：

- 功能
- 接口
- 诊断
- 数据
- 标定
- 非功能
- 约束

## 4. CM 符合性矩阵

CM 矩阵固定字段：

| 字段 | 说明 |
| --- | --- |
| 上游需求ID | 客户/系统/规格书需求编号 |
| 上游需求描述 | 保留来源语义 |
| 上游需求类型 | 功能、非功能、数据、权限、约束等 |
| 软件需求ID | `SWR-*` |
| 软件需求描述 | 软件层需求 |
| 分解关系 | 直接承接、拆分、合并、派生、约束继承、不适用 |
| 需求类型 | 功能、接口、诊断、数据、标定、非功能、约束 |
| 验证方式 | 评审、静态分析、单元测试、集成测试、台架测试、HIL测试、实车测试、待确认 |
| 覆盖状态 | 已覆盖、部分覆盖、未覆盖、待确认、不适用 |
| 一致性问题 | 缺参数、缺边界、冲突、不可验证、来源不明等 |
| 备注 | 其他说明 |

## 5. 枚举

分解关系：

- 直接承接
- 拆分
- 合并
- 派生
- 约束继承
- 不适用

覆盖状态：

- 已覆盖
- 部分覆盖
- 未覆盖
- 待确认
- 不适用

验证方式：

- 评审
- 静态分析
- 单元测试
- 集成测试
- 台架测试
- HIL测试
- 实车测试
- 待确认

## 6. 一致性问题

以下情况必须记录为一致性/不完整性问题：

- 上游需求缺少阈值、时间、状态、信号、接口对象或边界条件。
- 上游需求之间存在冲突。
- 上游需求无法验证，例如只写“友好”“高效”“稳定”。
- 软件需求无法判断是否完整覆盖上游需求。
- 由拆解产生派生需求，但派生依据需要业务或系统工程确认。

## 7. 设计输入集

设计输入集供后续方案设计读取，描述软件设计必须关注的能力、边界、约束和风险。它可以提示模块归属和交互方向，但不得替代软件方案设计，也不得提前写死代码结构。

## 8. 禁止事项

- 禁止把“前端开发”“后端开发”“数据库开发”“接口开发”等任务描述当成 `SWR-*` 主体。
- 禁止在没有上游依据时新增看似确定的软件需求。
- 禁止把待确认参数写成确定阈值。
- 禁止跳过 CM 矩阵。
- 禁止只给覆盖结论，不给软件需求和验证方式。
