# 代码分析规则

## 扫描顺序

1. 技术底座：README、包管理/构建文件（package.json、requirements.txt、pyproject.toml、go.mod、pom.xml、Cargo.toml、*.csproj 等），确定技术栈和启动方式。
2. 目录结构：顶层目录职责、模块边界。
3. 入口与路由：main/index/app 入口、路由与 API 定义。
4. 数据模型：数据结构、ORM 模型、数据库迁移。
5. 调用关系：核心调用链、数据流、控制流。
6. 测试与配置：测试覆盖、配置项、环境依赖。

可先用 `scripts/扫描代码结构.py` 得到技术栈标志文件、目录树和入口线索，但脚本输出只是线索，结论仍需读代码确认。

## 事实与推断判据

- 确定事实：代码中明确存在，可点到文件、符号、行为。必须给代码依据。
- 合理推断：从命名、调用、条件组合推断的业务含义。必须标「推断」。
- 待确认项：代码无法证明、需要业务或研发确认。进待确认清单。

判据：**能不能点到具体代码？** 能 → 确定事实（给依据）；只能猜含义 → 推断；连猜的依据都不足 → 待确认。

## 代码依据要求

- 每条确定事实必须给出文件路径，能精确到符号/函数更好；结构化 JSON 中必须写为 `{内容, 代码依据}` 对象，不得只写纯文本事实。
- 未读到的区域不得下结论，标为待确认项，写明「未读」。

## 两种运行语境

| 语境 | 重点 | 禁忌 |
| --- | --- | --- |
| 需求驱动 | 现有代码如何承接需求：可复用/需新增/需修改 | 不脱离真实代码空谈方案 |
| 代码驱动 | 当前系统真实做了什么 | 不把当前实现直接当成最优设计；不把推断写成事实 |

## 与代码生成闭环的衔接

- 在 `implementation-executor` 之后运行时，以 `implementation-record.json` 的「已实现且测试通过」任务对应代码为事实来源。
- 沿用实现记录的需求编号，保持需求—代码—文档追溯链。
- 实现记录 `可进入代码驱动出文档` 非 true 时，报告须标注「代码未全部验证，分析仅供参考」。

## 禁止事项

- 不在未读代码时下设计结论。
- 不把推断包装成确定业务规则。
- 不把 `codebase-analysis.json` 写入 `outputs/`（它是内部契约，进 `state/structured/`）。
- 分析报告不臆造代码里不存在的模块、接口或字段。
