# 代码实现执行规则

## 实现顺序

- 依赖优先：被依赖的模块/接口先实现。
- 可测试优先：能独立验证的单元先落地，尽早跑通测试。
- 高风险任务优先暴露：任务计划标为风险/预研的任务提前动手，尽早发现阻塞。

## 复用优先

- 动手前先查现有代码是否已有可复用实现，优先复用/扩展，不重复造轮子。
- 遵循项目既有目录结构、命名与风格；有 `knowledge/编码规范.md` 时以其为准。

## 测试门槛

- 每个任务必须有对应测试，覆盖其追溯的验收标准。
- 「测试通过」指测试真实运行且全部通过，不含：跳过（skip）、注释掉、删除断言、改测试迁就实现。
- 测试无法通过时，如实标记 `已实现未验证` 或 `未实现（阻塞）`，并写明原因。

## 可追溯

- 每个开发任务必须挂至少一个需求编号（FR-xxx 或规格书条目）。
- 无法追溯到需求的实现，列入待确认项，说明为何需要它。

## 状态枚举判定

- `已实现且测试通过`：代码 + 测试都到位且通过。
- `已实现未验证`：代码写了，测试缺失/未跑。
- `部分实现` / `未实现（阻塞）`：据实标注，阻塞项写清缺什么。

## 与代码驱动模式的交接

- 本 skill 产出「测试通过的代码 + implementation-record.json」后，才允许触发代码驱动出文档。
- `codebase-analysis-reporter` 以真实代码为事实来源，`detailed-design-writer` / `strategy-document-writer` 据此产出忠实文档。
- 详细设计/策略文档的**权威版本以编码后、测试通过的这一版为准**；编码前的前向设计只作为实现输入。
- 只有 `可进入代码驱动出文档 = true`（无失败、无阻塞）时，下游才应基于本次代码反出文档；否则先补齐实现与验证。

## 禁止事项

- 不把未实现功能标为已完成。
- 不把跳过/删除测试当作测试通过。
- 不把代码写入 `outputs/`（代码进仓库，`outputs/` 只放文档交付物）。
- 不臆造代码里不存在、需求里也没有的能力。
