# 样本文档结构总结

本文档根据以下样本整理：

- `PM2.5无效值优化策略文档_r01.docx`
- `通话自动降风量策略文档_r01.docx`
- `高速吹面防窜风策略文档_r01.docx`
- `循环换气策略文档_r01.docx`
- `防霉菌模式策略文档_r01.docx`

## 共同骨架

所有样本文档都包含以下固定前置部分：

1. 封面
2. 修订记录
3. 目录

正文的稳定主章节基本都是：

1. 概述
2. 信号描述
3. 逻辑设计
4. 附录

## 一级章节规律

### 1 概述

常见二级节：

- `1.1 需求描述`
- `1.2 模型信息`
- `1.3 功能概述`（部分文档有）

### 2 信号描述

最常见是：

- `2.1 输入信号`
- `2.2 输出信号`

有些功能会再增加：

- `2.3 档位映射`
- `2.3 状态定义`
- `2.3 参数映射`

### 3 逻辑设计

这是差异最大的部分，但仍有明显模式：

- 先给 `3.1 总体框架`
- 再按功能块拆分进入条件、执行动作、状态保持、退出条件
- 复杂功能会增加时序阶段、状态机、异常处理、共用约束

高频二级节：

- `开启条件`
- `进入条件`
- `执行动作`
- `退出条件`
- `特殊场景处理`
- `状态反馈`
- `共用约束条件`
- `延时判断及时序`

高频三级节：

- `使能条件`
- `状态判定`
- `模式判定`
- `前/后通道分别处理`
- `计时逻辑`
- `滤波处理`
- `故障处理`

### 4 附录

附录大多是定义类或汇总类表格：

- 状态机定义
- 档位定义
- 信号值定义
- 参数配置说明
- 时序说明

## 常见表格类型

### 封面审批表

- 4 行 2 列
- 常见字段：`编制 / 校核 / 会签 / 批准`

### 修订记录表

- 列通常是：`NO / 修订内容 / 修订人 / 版本号 / 发布日期`

### 模型信息表

- 列通常是：`编号 / 模型名称 / 描述 / 模型版本 / 模型执行周期 / 备注`

### 信号表

常见列：

- `序号 / 信号名 / 中文含义 / 取值范围 / 精度/单位 / 备注`
- 部分文档没有 `精度/单位`，只有 5 列

### 条件/状态类表

高频列：

- `序号 / 条件 / 说明`
- `状态 / 描述 / 说明`
- `项目 / 内容`
- `时间节点 / 动作 / 信号状态`

## 内容来源分工

样本文档体现出一个稳定分工：

- 需求文档负责：背景、目标、业务约束、用户设置、边界条件
- 代码/头文件/diff 负责：信号名、枚举、阈值、时序、状态机、执行动作
- 文档作者负责：把两者映射成可评审的工程表达

## 新文档建议

生成新策略文档时，优先按以下顺序组织内容：

1. 先写概述，说明“为什么要有这个功能”
2. 再写信号，说明“系统依赖什么输入、产生什么输出”
3. 再写逻辑设计，说明“在什么条件下做什么事情”
4. 最后在附录里放定义、映射、状态机、参数表

如果功能复杂，优先分块，不要把所有条件和动作堆成一段长文。
