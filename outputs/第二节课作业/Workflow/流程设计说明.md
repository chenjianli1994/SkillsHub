# 需求分析与规格书生成流程

## 1. 流程目标

把一段原始汽车软件需求转换为有知识依据的分析结果或待确认规格书。流程强调先诊断、后规格化；不能以完整排版掩盖 P0/P1 未确认。

## 2. 输入字段

| 字段 | 类型 | 必填 | 默认值 | 处理规则 |
| --- | --- | --- | --- | --- |
| raw_requirement | 文本 | 是 | 无 | 原始需求或补充说明 |
| confirmation_answers | 文本 | 否 | 空 | 对 analysis 清单的确认、冲突裁决或补充事实 |
| run_mode | 枚举 | 否 | full | 允许 `analysis` 或 `full` |
| allow_low_risk_assumptions | 布尔 | 否 | true | 仅允许 P2/P3 显式标注假设推进 |

## 3. 节点顺序

```text
用户输入
  -> 知识检索
  -> 需求分析
  -> 需求规格化
  -> 质量检查
  -> 结果输出
```

DEAP 画布实际配置为 7 个节点：用户输入、知识检索、需求分析、需求规格化、质量检查、结果输出，以及结束节点；结束节点用于收束输出，不改变业务处理顺序。

## 4. 分支规则

- `analysis`：完成知识检索、事实/推断/假设分类、问题诊断和确认清单后输出，不强行生成已确认规格书。
- `full`：先完成分析，再合并 `confirmation_answers`。P0/P1 未覆盖时，需求条目标为待确认并输出待确认草稿；已覆盖部分可以进入已确认需求。
- `allow_low_risk_assumptions=true`：只允许 P2/P3 使用假设，不改变 P0/P1 门禁。
- `run_mode` 非法：返回输入校验错误和允许值，不继续生成业务结论。

## 5. 质量检查

检查三类信息区分、知识来源、需求编号、功能字段、验收标准、冲突留痕、P0/P1 状态和参数禁编。失败时输出阻塞项和修正建议。
