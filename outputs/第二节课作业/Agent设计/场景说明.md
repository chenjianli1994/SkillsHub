# 汽车软件需求助手场景说明

## 1. 场景名称

汽车软件需求分析与规格书生成：动力电池冷却控制脱敏示例。

## 2. 服务对象

汽车软件产品工程师、系统工程师、软件开发工程师、测试工程师和需求评审人员。

## 3. 业务问题

动力电池冷却控制需求通常来自会议纪要、故障描述或跨团队讨论，容易出现阈值、时间、信号语义、故障等级和恢复条件没有确认，或者不同材料对同一规则表达不一致的问题。助手需要先诊断材料，再按确认状态生成可追踪规格书。

## 4. 核心任务

- 提取原文明确事实、合理推断、待确认假设。
- 诊断缺失、歧义、冲突、不可验证和范围不清问题。
- 对 P0/P1 未确认项建立确认清单；对 P2/P3 按条件显式标记假设。
- 生成 FR、DR、PR、NFR 编号，补充验收标准、来源依据、验证方式和追踪状态。
- 对传感器异常只描述已知处理意图，禁止编造温度、延时、信号名、故障等级或降级参数。

## 5. 输入与输出

输入字段：`raw_requirement`、`confirmation_answers`、`run_mode`、`allow_low_risk_assumptions`。

输出固定包含：执行摘要、确定事实、合理推断、待确认假设、问题诊断、确认清单、需求规格书或待确认草稿、需求追踪矩阵和质量检查结论。

## 6. 边界

本 Agent 不替代安全工程师、系统架构评审或实车/HIL 验证；不从通用常识推导本项目未提供的控制参数；不把平台运行结果写成汽车产品的已批准安全结论。
